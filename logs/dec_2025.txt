@16:00 countAllPathsDFS

The tricky bit: why is visited marked as false at the end?
It's backtracking.
---
The problem: Count all possible unique paths from a source
node to a destination.
---
This algorithm works recursively, with the stack of "DFS"
being implicit via the call stack. It starts at the source,
then iterates the adjacent nodes, seeing how many paths
are from that node to the destination. It ends with the base
case of the source being the destination (that's one path).
It also ends when all the nodes are traversed. It then unwinds.
---
The reason why it unmarks a visited node is that has to do
with the "unwinding" logic, kind of like a car backing up
and trying another road. It's possible for those vertices
to be re-used again.

of the periods, and ensuring that there are no scheduling conflicts.
---
First to address: How to make sure class A appears at least once?
The notation used here is important. $X_{A1}$ means "class A
schedule in period 1".
that's (A in period 1 or period 2), a disjunction operation,
or $X_{A1} + X_{A2}$. But, we also need to make sure they both
can't exist. In other words, it CAN'T be A in period 1 AND
in period 2. That's a conjection that's been negated, or
$\overline{X_{A1}X_{A2}}$. Both these terms need to be multiplied
together to get $(X_{A1} + X_{A2})(\overline{X_{A1}X_{A2})}$.
Repeat for every class.
---
Now, to address the conflicts. If classes E and B can't be
scheduled at the same time. That means, E and B can't be
in period 1 together, and E and B can't be in period 2 together.
Those are 2 negated conjunctions. Using the notation previous
defined: $(\overline{X_{E1}X_{B1}})(\overline{X_{E2}X_{B2}})$.
Repeat for every conflict.
---
De Morgan's laws are required to get some of the negated bits
into CNF form, otherwise it is pretty much set up for CNF.
---
Determining satisfiability case would probably involve
finding a combination where there aren't any scheduling
conflicts, then filling out the rest. I'm sure doing it that
way cuts down on all the possibilities to try out.

@15:54 5.4.4 equivalent expressions using NOR and NAND
#! dz $/5_4_4
Problem is to convert some multiply/add arithmetic to use
only NAND operations, then only NOR operations
---
these problems suck. Either you're stuck writing a long
equation that is error prone, or you just happen to see
a small shortcut that isn't covered in the textbook. Either
way, it's bookkeeping.
---
First things first, some identities need to be defined
from first principles.
---
NOT from NAND. Negation in NAND is NAND-ing itself, NAND(A, A).
---
AND/multiply from NAND and NOT.
After getting NOT from NAND, you can get multiplication, which
is just binary AND, by negating a NAND:
XY = AND(X,Y) = NOT(NAND(X, Y)) = NAND(NAND(X, Y), NAND(X, Y)).
---
Add/OR from NAND, NOT, and De Morgan's law. This one is tricky
to derive under pressure, so pay attention. getting OR(x, y),
or X + Y with a NAND requires use of De Morgan's laws. However,
those only work with negation. So, the expression needs
to be double negated like so NOT(NOT(X + Y)). From there
the inner loop can be converted to multiplication using
De Morgan's Law, using AND for boolean multplication:
NOT(AND(NOT(X), NOT(Y)))). Look at that, there's a NOT and
an AND in the outer layer. It becomes NAND(NOT(X), NOT(Y)),
which becomes NAND(NAND(X, X), NAND(Y, Y)).
---
Notice the paterns in NAND. AND in AND is XYXY, OR is XXYY.
---
NOT from NOR. It ends up being a similar thing to NAND:
NOR(A, A). Use the truth tables to confirm this.
---
OR/addition from NOR. Negating a NOR yields OR, or boolean
addition: OR(X, Y) = NOT(NOR(X, Y)) = NOR(NOR(X, Y), NOR(X, Y)).
---
AND/addition from NOR via DM and NOT. De Morgan's law again, the inverse of
OR from NAND: AND(X, Y) = NOT(NOT(X, Y)) = NOT(OR(NOT(X), NOT(Y)) =
NOR(NOT(X), NOT(Y)) = NOR(NOR(X, X), NOR(Y, Y)).
---
Again the patterns: OR with NOR is XYXY, AND with NOR is XXYY.
---
Identities derived, my process for getting these into NAND
and NOR was an iterative top-down approach. I addressed
the arithmetic operations first, then did the negation last.
Each time I rewrote a version of the equation. I pretended
I was a compiler, and used dots to keep track of where I was.
It stopped being a meaningful expression very quickly, and just
turned into a substitution problem with bookkeeping.

@16:40 6.2.4 Identifying properties of relations, continued
#! dz $/6_2_4
You have finite sets of elements for a domain, and then
a relation. The goal is to determine reflexsive, symmetric,
and transitive properties for each.
---
I liked using abbreviations for these answers.
Reflexive (R), Anti-reflexive (AR), not reflexive (NR),
symmetric (S), anti-symmetric (AS), not symmetric (NS),
transitive (T), not transitive (NT).
---
Reflexive was pretty easy to pick out. You just need to see
if there are any pairs of themselves. If there aren't any,
it's AR. If there's some, it's NR. If there's one for every
element, it's R.
---
Symmetric is about going through each pair where the
elements are different, and seeing if the reverse is there
as well. Similar pattern to R. No matches: AS, some matches NS,
all matches: S.
---
Transitive took me a while longer to grok, because I wasn't
confident on what it meant for every element to be satisified.
Transitive doesn't mean that every element is transitive
with every other element, which is the kind of property that
R and S had. Instead, the process goes kind of like this:
Find a pair (a, b). If there is any pair where the first
value is b, like (b, c), there must be a corresponding (a, c).
---
If there is (a, b) but no pair where b is first, that
doesn't disqualify it from being transitive I think?
---
Tricky thing. If there's an (a, b) and a (b, a), there needs
to be an (a, a). Transitive property is if (x, y) and (y, z),
then (x, z). x and z happen to both be a, which is okay.
in fact, x, y, and z can all be the same, so a relation with
the pair (a, a) is both reflexive and transitive (it is not
symmetrical, those need to be different, otherwise every
reflexive function would be symmetric).
---
So this is a pretty good way to warm up intuition on
these. They were small enough that I could ask claude for
some help on these.
