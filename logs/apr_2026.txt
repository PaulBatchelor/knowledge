@14:48 setting up ugen interface: initial thoughts #timelog:00:14:49
#! dz $
It's now time to build out the standard means for creating
re-usable unit generators.
---
Each ugen will be associated with a 1-3 character string called
a symbol (A-Z letters only). 'SIN' could be the symbol for
a sine oscillator. A symbol is converted to a 16-bit key,
and this key is used to find an associated function (via
an index). This function is the ugen's initialization function.
The 16-bit key has an extra bit. When flipped, turns into a key
holding the ugen's compute function.
---
Here's how I think it should work in practice. The API pushes
a symbol to the system stack (not the parameter stack), which
internally turns the symbol into a 16-bit key. Then the API
calls some kind of "create ugen" function, passing
along a pointer to the current context. This function
pops the key off the stack, and uses that to retrieve the
init function for that particular ugen. The key is pushed
back onto the stack, and this init function is called. Inside
this init function, memory is allocated for ports and state.
The key is popped off again, the extra-bit is flipped to
get the index of the compute function, which is stored inside
the ugen data. The ugen address is then pushed onto the system
stack. Most likely, that value would get put onto a list somewhere.
