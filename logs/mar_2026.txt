@10:15 check-in
#! dz thoughts/projects/sonilo/tasks/allocator
Here's the vision of memory allocation in Sonilo as it stands
today.
---
Sonilo is used a fixed-size region of memory that is $2^{16}$
32-bit words. These words are formed into 64-word groupings
known as blocks.
---
In theory, there are 1024 available blocks, and these
can be addressed in a 10-bit namespace. A free list is used
to manage these blocks (this takes up some memory). An additional
operation is used to convert this 10-bit block address to a
16-bit word address. This operation skips over the blocks used
for the free list.
---
The name of the game after this is to imagine structures which
can fit on a block. Blocks can be used as arrays which can
be used as stacks. A block can be used as a zero page for
storing temporary information. There is also the notion of
a "bit set", a data structure designed to be used to hold
a set of block addresses, with the idea that allocated blocks
could be grouped together and managed. Blocks are also the
building block for the audio rendering system. A block can
be treated as a 64-sample block of audio in a block-based
rendering system.
---
The next level of concern is allocating regions of memory
that are smaller than a block, which is a requirement for most DSP
algorithms that need some kind of state. For this, a block
of memory is divided up using a buddy system. This allows allocating
a power-of-two chunk of memory inside of a block. The buddy
system implemented in Sonilo uses word 6-bit word offsets.
Presumably, these values would be added to some base word
address of a block.
---
Another system (not yet built at this time) is built on top
of the buddy system, known more or less
as the "buddy slot" allocator. The goal of this abstraction
is to provide a interface similar for malloc/free for allocating
small regions of memory. This allocator manages
blocks from the free list and instances of buddy allocators.
An allocator would take some memory request, and turn that
into arguments for an instance of a buddy allocator.
The free operation would take a word address, and attempt to find
a buddy instance associated with it.

@20:41 memstack: a convenient interface for managing blocks #ink:09i #timelog:00:51:04
#! dz thoughts/projects/sonilo/tasks/allocator
Working name "memstack". I realized that allocating blocks
relied on a few components working together, and wanted
to work that out a bit better.
---
It's not enough to be able to allocate a block and free that
block. There needs to be a way to allocate many blocks, and
then free some or all of those blocks. At the bare minimum,
there needs to be a distinction between blocks that have
an indefinite lifetime (such as those in a patch), and those
that have a temporary lifetime (scaffolding used to construct
patches). Also, collections of blocks should be able to be
freed in one bulk operation, not individually.
---
The "memstack" interface consists of a "free list" (what
I previously referred to as a block list), a set data structure
(implemented using bitmaps) to keep track of blocks being used,
and a stack data structure. Two sets are used, a "main" set used
for persistent memory, and "temp" set used for initial scaffolding.
4 individual components, each with memory addresses, that
can be stored in 2 words.
---
Allocating blocks is a matter of allocating a block and storing
it in a set. You can do this in a single operation, where the
input word contains the address of the free list and the
address of the set to store the block address in.
---
Switching between the temp set and the main set could be
generalized as a swap operation between two words, or a double
word if the words are continguous. So, I'm thinking about
building some general swap operations to help with this
task.
